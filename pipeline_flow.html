Stage 1 · parse_query_node
🔍 Parse Query
Builds a full AST from the SQL. Extracts tables, columns, joins, filter predicates, GROUP BY, ORDER BY, CTEs, subqueries, and SELECT *. Never raises — always returns what was parseable.

Stage 2 · fetch_metadata_node
🗃️ Fetch Metadata
For every table.column in the parsed query, performs an exact-match DuckDB lookup. Calculates metadata_coverage score (found columns ÷ total referenced columns).
duckdb LOWER() exact match
→ retrieved_metadata + coverage score

Stage 3 · fetch_explain_node
⚡ Fetch EXPLAIN Plan
Runs EXPLAIN LEVEL=2 on the Impala cluster. Parses the text plan into structured PlanNode objects: operator, scan bytes, join strategy, warnings.
⚠️ Skipped in --offline mode

→ ExplainPlan

Stage 4 · rule_lint_node
📋 Rule Lint (8 Rules)
Runs all 8 deterministic rules independently. Each returns list[Finding]. Rules never raise — missing data silently returns [].
R001 SELECT * R002 Partition filter R003 Non-sargable R004 Cross join R005 ORDER BY limit R006 Missing stats R007 Broadcast join R008 PII unmasked

Stage 5 · llm_analyzer_node
🤖 LLM Analyzer
OpenAI GPT receives AST summary + plan + lint findings + metadata sample. Runs a tool-call loop (max 4 iterations) to look up any additional facts before producing its analysis. Never invents schema facts.
Tools available in loop:
· lookup_column_metadata(table, column)
· get_table_stats(table)
· run_explain(sql)
langchain-openai langgraph langchain-core tenacity duckdb
→ analyzer_messages (conversation)
analysis messages
Stage 6 · rewrite_proposer_node
✏️ Rewrite Proposer
Uses structured output to produce a validated list of SQL rewrite candidates. Each rewrite targets specific rule IDs from the lint findings and must include a rationale.
langchain-openai with_structured_output pydantic v2
→ CandidateRewriteList

Stage 7 · validator_node
✅ Validator
For each candidate: (1) checks AST structural equivalence — rejects if tables are dropped. (2) Re-runs EXPLAIN on candidate SQL. (3) Computes PlanDiff — only marks verified=true when verdict is IMPROVED. Rejected candidates are retried once.
⚠️ Rewrites marked UNVERIFIED in --offline mode
sqlglot.diff impyla pydantic v2
→ list[ValidatedRewrite]
validated rewrites

📊 Build Report
Assembles the final ReviewReport: merges lint findings with validated rewrites, extracts PII flags, computes metadata_coverage, assigns verified status to each issue.

→ ReviewReport (JSON + CLI)

LangGraph Orchestrates Everything — Including Deterministic Stages
2 of 8 nodes touch the LLM. The other 6 are fully deterministic.
LangGraph StateGraph
│
├── parse_query_node 100% deterministic · sqlglot
├── fetch_metadata_node 100% deterministic · duckdb lookup
├── fetch_explain_node 100% deterministic · impyla call
├── rule_lint_node 100% deterministic · 8 rule functions
│
├── llm_analyzer_node 🤖 LLM · OpenAI GPT + tool loop
├── rewrite_proposer_node 🤖 LLM · structured output
│
├── validator_node 100% deterministic · sqlglot.diff + impyla
└── build_report_node 100% deterministic · pydantic assembly
Why LangGraph for deterministic nodes?
All nodes share a single AgentState TypedDict — LangGraph passes it automatically between stages. The conditional retry edge (validator → rewrite_proposer) would need manual routing logic without it. Each node is just a plain Python function (state) → dict — LangGraph handles the wiring.
